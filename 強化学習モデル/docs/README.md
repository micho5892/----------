


# LBM-RL HeatSink Optimizer

本プロジェクトは、Taichiベースの格子ボルツマン法（LBM）シミュレータと強化学習（Stable-Baselines3）を統合し、最適なヒートシンク形状を自動生成するシステムです。トップフロー冷却環境下において、圧力損失を抑えつつ熱輸送量（Nu数）を最大化する、3Dプリンターで造形可能な有機的形状を探索します。

---

## 1. 現状の仕様

### 1.1 物理シミュレーションモデル
*   **ソルバー**: Taichiを用いたGPUアクセラレーションLBM（D3Q19モデル）。
*   **流体-固体連成**: 体積ペナルティ法（VPM）を利用。シミュレーションを止めることなく、固体率（$\phi$）および符号付き距離場（SDF）の配列を書き換えるだけで瞬時に形状の変更が可能です。
*   **境界条件（トップフロー想定）**:
    *   **入口 (Inlet)**: 領域上部（天井）からの一定流速による冷風の吹き込み。
    *   **出口 (Outlet)**: 領域側面四方からの流出。
*   **熱源モデル**:
    *   ベースプレート（Z=0〜5層）は「削ることのできない固定熱源（等温壁）」として保護されます。
    *   その上部の領域が「削れる熱伝導体（フィン）」として設定されており、現実のCPUクーラーの物理的振る舞いを模倣しています。

### 1.2 強化学習環境 (`LBMHeatSinkEnv`)
Gymnasium APIに準拠したカスタム環境として実装されています。

*   **状態空間 (Observation)**:
    *   空間全体の3Dテンソル `[5, nx, ny, nz]`。
    *   チャネル構成: 速度ベクトル($V_x, V_y, V_z$)、温度($T$)、符号付き距離場($SDF$)。
*   **行動空間 (Action)**:
    *   連続値4次元 `[X, Y, Z, R]`（値域 `[-1.0, 1.0]`）。
    *   仮想球（メタボール）の中心座標と半径を指定し、対象領域のSDFを減算（ブーリアン演算）することでヒートシンクを削り出します。
*   **報酬関数 (Reward)**:
    *   直前ステップからの「熱輸送量（Nu数）の維持・向上」および「圧力損失の改善量」の差分に基づいて算出されます。
    *   無操作を防止するためのタイムペナルティが含まれます。
*   **高速化機構**:
    *   **空振り判定**: エージェントがブロック表面に届かない無効な座標を指定した場合、LBM計算をスキップし即座にペナルティを与えます。
    *   **スナップショット＆リストア**: 初回エピソード時のみ定常状態までのウォームアップを行い、その状態をメモリにバックアップします。次エピソード以降はバックアップをリストアするだけで一瞬で初期化が完了します。

### 1.3 AIアーキテクチャ
*   **アルゴリズム**: PPO (Proximal Policy Optimization)
*   **特徴抽出器 (Feature Extractor)**: 巨大な3D観測テンソルを処理するため、PyTorchで定義したカスタム3D-CNNをPolicyネットワークのフロントエンドとして組み込んでいます。
*   **演算の分離**: LBMシミュレーションはGPU（Taichi）で、PPO/3D-CNNの学習はCPU（PyTorch）で処理することで、VRAMの競合と枯渇を回避しています。

## 2. 使い方

### 2.1 前提条件
以下のライブラリがインストールされている必要があります。
```bash
pip install taichi numpy sympy pyyaml pyvista
pip install stable-baselines3[extra]
pip install torch torchvision torchaudio  # (※GPU環境がある場合でも本プロジェクトのAI推論はCPUで動作可能)
```
※最適化モジュール等の依存のため、既存のLBMシミュレータコード（`refactoredv2.7`等）が同階層または適切なパスに配置されている前提です。

### 2.2 環境の動作テスト (ランダムエージェント)
強化学習を回す前に、シミュレータがエラーなく動作するか、形状の変更処理が適切に行われるかを確認するテストスクリプトです。

```bash
python -m 強化学習モデル.random_agent_test --U-inlet-p 0.1
```
*   実行完了後、カレントディレクトリに `rl_state_initial.vti` と `rl_state_final.vti` が生成されます。これらを ParaView 等で開き、形状や流場が正しく計算されているか確認してください。

### 2.3 強化学習の実行 (PPO)
AIにヒートシンクの形状を学習・設計させます。

```bash
python -m 強化学習モデル.train_agent --timesteps 10000
```
*   `--timesteps`: 学習させる総ステップ数（1エピソード=100ステップ）。本番の学習には `20000` 〜 `50000` 程度を推奨します。
*   学習中、`models/checkpoints/` ディレクトリにモデルのバックアップが定期保存されます。
*   学習完了後、最終モデルでテストプレイが行われ、`rl_state_trained_result.vti` が出力されます。

### 2.4 学習状況の可視化 (TensorBoard)
学習の推移（報酬の増加やLossの減少）はTensorBoardでリアルタイムに確認できます。別のターミナルを開き、以下を実行してください。
```bash
tensorboard --logdir logs/heatsink_tb
```

## 3. 変更・拡張ガイドと注意点

本モデルをカスタマイズする際の方法と技術的注意点です。

### 3.1 物理・流体パラメータの変更
`train_agent.py` や `random_agent_test.py` 内で `RLParameterOptimizer` を初期化している箇所を修正します。
```python
optimizer = RLParameterOptimizer(
    nx=64, U_inlet_p=0.5, L_domain=0.1, fluid="Water"  # 流体を水に変更、流速を0.5m/sに変更
)
```
オプティマイザが自動的にLBM空間の安定条件（緩和時間等）を逆算し、シミュレータに安全なパラメータを渡します。

### 3.2 形状変更アクションの拡張（カプセル減算など）
現在のアクション空間は「球」ですが、より複雑な形状（スリットや柱）を効率よく学習させるために、「始点・終点・半径」を持つカプセル（線分）で削る拡張が有効です。
1.  `lbm_heatsink_env.py` の `action_space` の次元を4から7に変更。
2.  `shape_modifier.py` に `subtract_capsule` カーネルを追加実装。
3.  `step` メソッド内のアクション展開部を書き換える。

### 3.3 報酬関数のチューニング
AIが「ただの大きな穴を開けるだけ」など、意図しない形状（Reward Hacking）を生成する場合は、`lbm_heatsink_env.py` の `step` メソッド内にある報酬式を調整してください。
```python
# 例：圧力損失の改善より、Nu数（冷却力）の維持を強く評価させたい場合
nu_reward = (current_nu - self.prev_nu) * 5.0          # 係数を上げる
pressure_improvement = (self.prev_pressure_drop - current_pressure_drop) * 10.0 # 係数を下げる
```

### 3.4 【重要】TaichiとPyTorchのVRAM競合について
現状、Taichi（流体計算）はGPUのVRAMを確保します。もしPyTorch（AI側）も `device="cuda"` でGPUを使用するように設定変更した場合、VRAMの取り合いによる `CUDA out of memory` エラーが発生する可能性があります。
その場合は、`simulation_runner.py` 内のTaichi初期化部分でVRAMの使用上限を制限してください。
```python
# simulation_runner.py
ti.init(arch=ti.gpu, device_memory_fraction=0.6) # Taichiの確保量を60%に制限
```

### 1. `rollout/`（AIのシミュレーション上の成績表）
AIが実際に環境で行動してみて、どうだったかという結果です。人間が一番気にするべきところです。

*   **`ep_len_mean` (100)**
    *   **意味:** 1エピソードが平均何ステップで終わったか。
    *   **現状:** 今回は100ステップで強制終了（または途中でブロックが無くなる等の終了条件）させているため、常に100になります。
*   **`ep_rew_mean` (-12.8) 🌟重要！**
    *   **意味:** 1エピソード（100回の行動）でもらえた**報酬の平均合計点**です。
    *   **見方:** この数値が**右肩上がりに増えていれば、AIは賢くなっています。** 今回は空振りペナルティや時間経過のマイナスがあるため負の値ですが、学習初期の`-300`などから比べると飛躍的に改善しており、AIが「風の通り道を上手く作る方法」を見つけ始めている証拠です。

### 2. `time/`（学習の進捗とスピード）
学習にかかっている時間や進み具合です。

*   **`fps` (0)**
    *   **意味:** 1秒間に何ステップ学習できたか（Frames Per Second）。
    *   **現状:** 普段のゲームAI等なら数百〜数千になりますが、今回は重い流体シミュレーション（1ステップあたり約1秒）を回しているため、1未満となり `0` と表示されています。異常ではありません。
*   **`iterations` (10)**
    *   **意味:** PPOアルゴリズムが「行動データを集める → AIの脳（ネットワーク）を更新する」というサイクルを回した回数です。
*   **`time_elapsed` (1334)**
    *   **意味:** 学習開始からの経過時間（秒）。約22分経っていることがわかります。
*   **`total_timesteps` (1000)**
    *   **意味:** これまで環境で行動した累計ステップ数。

### 3. `train/`（AIの脳内・学習の健康診断）
ニューラルネットワークの学習状況です。複雑な数値が並んでいますが、**見るべきポイントは3つ**です。

*   **`explained_variance` (0.429) 🌟超重要！**
    *   **意味:** AIが「自分がこういう行動をとったら、将来どれくらい報酬がもらえるか」を**どれだけ正確に予測・理解できているか**を示します。1.0が完璧な理解、0.0以下は全く分かっていない状態です。
    *   **見方:** 最初はマイナスや0付近をうろつきますが、これが**プラス（今回は0.429なので42%理解している！）**になってきたら、AIが「適当に削る」のをやめて、「ここを削れば風が通ってスコアが上がるな」と**因果関係の法則を掴んだ**という大成功のサインです。
*   **`value_loss` (0.483) 🌟重要**
    *   **意味:** AIの「もらえる報酬の予測予測」が、現実とどれくらいズレていたか（誤差）です。
    *   **見方:** 以前のバグがあった時はこれが `1,600,000` などになっていましたが、今は `0.4` 程度に収まっています。**この値が低く安定している限り、AIはパニックにならずに健康に学習できています。**
*   **`entropy_loss` (-5.68)**
    *   **意味:** AIの「行動のランダムさ（探索度合い）」です。
    *   **見方:** 学習の最初は色々な場所をランダムに削って試すためエントロピーが高く、学習が進んで「ここを削れば正解だ」と自信を持つようになると、この数値は徐々に小さく（マイナス方向に大きく）なっていきます。
*   **`approx_kl` (0.025) / `clip_fraction` (0.213)**
    *   **意味:** 脳を一度に急激にアップデートしすぎないようにする安全装置（PPOの特徴）の作動状況です。
    *   **見方:** 正常な範囲で機能していることを示しています（気にしなくて大丈夫です）。
