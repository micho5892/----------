# データ主導モジュール化リファクタリング — タスク一覧（優先順位順）

このドキュメントは、設計書「データ主導のモジュール化設計」に基づく `main_v2 copy.py` の修正内容と、ファイル分割方針を、**タスクごとに優先順位を付けて**整理したものです。

---

## 優先度：高（基盤づくり）

### タスク1: SimulationContext の導入
- **目的**: すべてのデータを一つの「箱」にまとめ、共通インターフェースにする。
- **内容**:
  - `SimulationContext` クラスを新設する。
  - 保持する Taichi field: `cell_id`, `sdf`, `f`, `g`, `rho`, `vel`(または`v`), `temp`。
  - 格子サイズ `(nx, ny, nz)` はコンストラクタで受け取り、ここで全 field を確保する。
- **完了条件**: 他モジュールが「ctx を渡すだけ」でデータにアクセスできる状態になること。

### タスク2: セルID（タグ）マップの導入
- **目的**: 境界・形状の判定を「位置のハードコード」から「ID」に移す。
- **内容**:
  - ID を定数または Enum で定義する（例: FLUID_A=0, SOLID=10, SOLID_HEAT_SOURCE=11, INLET=20, OUTLET=21）。
  - 現行の `mask` を廃止し、役割をすべて `cell_id` に集約する。
  - 形状構築では `mask` の代わりに `cell_id` に上記 ID を書き込む。
  - 入口・出口は座標範囲で `cell_id` を上書きする仕組みを用意する。
- **完了条件**: 境界判定が「k == nz-1」などに依存しないこと。

### タスク3: ファイルを役割ごとに分割する
- **目的**: 保守性を上げ、設計書のモジュール構成に合わせる。
- **方針**: クラスごとではなく**役割（モジュール）ごと**にファイルを分ける。
- **分割案**:
  - `context.py` — SimulationContext、セルID定数
  - `config.py` — SimConfig、materials_dict の組み立て
  - `d3q19.py` または `lbm_constants.py` — D3Q19、LBM定数
  - `geometry.py` — GeometryBuilder、build_* 系の @ti.kernel
  - `boundary.py` — BoundaryConditionManager と境界用 kernel
  - `solver.py` — LBM_Solver（collide / stream / update_macro）
  - `export.py` — VTKExporter、GIF用（必要なら）
  - `main.py` または `run.py` — run_simulation、メインループ
- **完了条件**: 上記ファイルが存在し、import でつながって一通り実行できること。

### タスク4: Geometry モジュールの分離
- **目的**: 形状・領域管理を一か所にまとめる。
- **内容**:
  - `GeometryBuilder`（または `Geometry`）クラスを新設する。
  - 現行の `build_karman_cylinder` 等を、`ctx.cell_id` と `ctx.sdf` を更新するメソッドに移す。
  - `set_inlet_outlet(ctx, inlet_box, outlet_box)` で入口・出口の `cell_id` を上書きする。
  - 将来用に `set_heat_source(ctx, target_surface_box)`、`load_stl(ctx, filename)` のシグネチャを用意する（中身は未実装可）。
- **完了条件**: 形状設定がすべて Geometry 経由で行われること。

---

## 優先度：中（境界・Solver の整理）

### タスク5: BoundaryConditionManager の分離
- **目的**: 境界条件を「ID だけ」で適用するようにする。
- **内容**:
  - `BoundaryConditionManager` クラスを新設する。
  - `apply_velocity_bc(ctx, target_id, vel_vec)` — cell_id == target_id のセルに速度境界を適用する。
  - `apply_heat_flux_bc(ctx, target_id, flux_val)` — 指定 ID に定熱流束を適用する。
  - 境界の「位置」はハードコードせず、`cell_id` の値だけで判定する。
  - 現行の `apply_bc` の内容を上記メソッドに移し、Solver からは `bc.apply_*` を呼ぶだけにする。
- **完了条件**: 入口・出口・熱源の設定がインデックスに依存しないこと。

### タスク6: LBM Solver の Context 受け渡し化
- **目的**: Solver を「状態を持たず、Context を引数で受け取る」形にする。
- **内容**:
  - `collide_and_stream(ctx: SimulationContext, materials_dict)` のようなメソッドを用意する。
  - 内部では `ctx.cell_id`, `ctx.f`, `ctx.g`, `ctx.rho`, `ctx.vel`, `ctx.temp` を参照する。
  - `materials_dict` から cell_id に応じた tau_f / tau_g を取得して計算する。
  - まずは流体 ID のみ対応し、固体はスキップでも可。
- **完了条件**: Solver が ctx と materials_dict だけに依存して動作すること。

### タスク7: Config & Material の整理
- **目的**: ID と物性を対応づけ、無次元パラメータの基準を管理する。
- **内容**:
  - 「ID → 物性」のマッピング（例: materials_dict）を SimConfig または MaterialConfig で保持する。
  - 代表長さが変わったときに無次元パラメータを再計算する入口を用意する（Geometry の get_hydraulic_diameter 等と連携）。
  - ID ベースの境界条件（どの ID にどの速度・熱流束を適用するか）を設定できるようにする。
- **完了条件**: 物性と境界条件が ID ベースで一覧・変更できること。

---

## 優先度：中〜低（出力・最適化）

### タスク8: VTK（VTI）エクスポートの追加
- **目的**: 3D 可視化・ParaView 連携を可能にする。
- **内容**:
  - `VTKExporter` クラスを新設する。
  - `export_step(ctx, step, filepath_template)` で ctx の rho, vel, temp, cell_id を VTI 形式で書き出す。
  - PyVista や meshio 等を利用する。
  - 既存の GIF 出力はレガシーまたはオプションとして残す（export_gif_frames 等に分離可）。
- **完了条件**: 指定ステップで .vti が出力され、ParaView で開けること。

### タスク9: メインループの整理
- **目的**: 設計書の疑似コードに沿った実行フローにする。
- **内容**:
  - ctx 生成 → Geometry で形状・入口出口設定 → Config/Materials 準備 → Solver / BC / Exporter 生成 → init → ループ（collide_and_stream → apply_velocity_bc → apply_heat_flux_bc → マクロ更新 → 必要ならパーティクル → export_step）。
  - Nu や局所値の計算（get_local_Nu, get_avg_grad 等）は ctx を引数に取るユーティリティにし、Solver/Boundary から切り離す。
- **完了条件**: メインスクリプトが「コンポーネントの組み合わせ」だけを記述していること。

### タスク10: その他の整理
- **目的**: 定数・パーティクルの扱いを方針どおりにする。
- **内容**:
  - パーティクルを SimulationContext に持たせるか、可視化用の別オブジェクトにするかを決める。
  - C_s や MRT の S 等を Config または D3Q19 側にまとめ、マジックナンバーを減らす。
- **完了条件**: 役割が明確になり、マジックナンバーが減っていること。

---

## 優先度：低（将来の最適化）

### タスク11: メモリ・計算量の最適化（任意）
- **目的**: 設計書で触れている「疎行列」「ダブルバッファ廃止」を検討する。
- **内容**:
  - Taichi Sparse（ti.root.pointer / bitmasked）の利用を検討する。
  - ダブルバッファを廃止し、In-Place Stream（A-A パターン）でメモリを削減する。
- **備考**: Context と ID ベース設計が固まってから実施する想定。

---

## 実施の進め方（推奨）

1. **Phase 1**: タスク1 → タスク2 → タスク4（Context と ID と Geometry の基盤）
2. **Phase 2**: タスク3（ファイル分割）を、Phase 1 の結果を反映しつつ実施
3. **Phase 3**: タスク5 → タスク6 → タスク7（境界・Solver・Config の整理）
4. **Phase 4**: タスク8 → タスク9 → タスク10（出力・メインループ・その他）
5. **任意**: タスク11（最適化）は必要に応じて後から
